## What Was Built

### 1. fetchNpmSource() with esbuild Transform — patches/index.ts

Downloads the npm package, extracts cli.js, transforms ESM→CJS via esbuild:
```
npm pack @anthropic-ai/claude-code@{version}
tar xzf ... package/cli.js
npx esbuild cli.js --bundle --format=cjs --platform=node --outfile=cli.cjs
```

Key details:
- esbuild v0.28.0 converts 753 static imports → `require()` calls
- `--platform=node` keeps Node builtins external
- `--bundle` resolves and transforms the import graph
- Output: ~17MB CJS file (vs 13.5MB ESM input — esbuild adds require() wrappers)
- Timeout: 120s for esbuild (the transform is CPU-intensive on 13MB)

### 2. import_meta.url Fix

esbuild creates `var import_meta = {};` as a stub for `import.meta` in CJS mode.
This causes `createRequire(import_meta.url)` → `createRequire(undefined)` → TypeError.

Fix: string-replace esbuild's empty stub with a proper polyfill:
```javascript
'var import_meta = { url: require("url").pathToFileURL(__filename).href };'
```

This provides a valid file URL for the `createRequire` call. The `__filename` comes from
the CJS function wrapper parameters.

### 3. CJS Function Wrapper

The esbuild-transformed CJS content is wrapped in Bun's module format:
```javascript
(function(exports, require, module, __filename, __dirname) {
  // ... esbuild CJS output ...
})
```

### 4. clearBytecode Pipeline — nativeInstallation.ts

- `rebuildBunData()`: accepts `clearBytecode` param

  - Overrides `encoding` from 1 (bytecode) to 0 (source interpretation)
  - Other 10 modules untouched
- `repackNativeInstallation()`: threads `clearBytecode` to `rebuildBunData()`
- `nativeInstallationLoader.ts`: threads param through dynamic wrapper

### 5. Raw Binary Overwrite — repackMachO()

**Critical LIEF bug bypassed**: `bunSection.content = newData` is silently ignored
for Mach-O unless `extendSegment()` is called first. When the new data is smaller
than the original (our case: 21MB vs 128MB), raw overwrite works:

```typescript

  fs.renameSync(tempPath, outputPath);
```

### 6. BUN_TRAILER-based Format Detection — extractBunDataFromSection()

The original format detection used a 4KB tolerance window:
`expectedLength >= sectionData.length - 4096`

This fails for zero-padded sections (21MB data in 128MB section = 107MB gap).
Replaced with BUN_TRAILER validation:

```typescript
function hasBunTrailerAt(sectionData, headerSize, bunDataSize) {
  const trailerStart = headerSize + bunDataSize - BUN_TRAILER.length;
  return sectionData.slice(trailerStart, trailerStart + 16).equals(BUN_TRAILER);
}
```

## Key Technical Lessons

1. **esbuild `--bundle --format=cjs` handles ESM→CJS** but creates an empty
   `import_meta` stub that must be replaced
2. **LIEF's Mach-O section writer is broken** — `section.content =` is silently
   ignored unless `extendSegment()` is called. Raw overwrite is the reliable path.
3. **Bun's module encoding field** — `encoding: 0` = interpret source text,
   `encoding: 1` = use bytecode. Clearing bytecode requires setting encoding=0.
4. **Zero-padded sections need trailer validation** — tolerance windows fail when
   the section is much larger than the data.
5. **The esbuild output has a fundamentally different code structure** than Bun's
   minified output. All binary patches need regex pattern migration.
