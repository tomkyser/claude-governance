## Root Cause Analysis

### Bun Binary Module Layout

The Bun binary stores modules in a `__BUN/__bun` Mach-O section. Each module has:
- `name`: StringPointer (offset, length)
- `contents`: StringPointer — the JS source code
- `sourcemap`: StringPointer
- `bytecode`: StringPointer — pre-compiled bytecode (111MB for cli.js!)
- `moduleInfo`: StringPointer
- `bytecodeOriginPath`: StringPointer
- `encoding`: u8 (1 = has bytecode)
- `loader`: u8 (1 = JS)
- `moduleFormat`: u8 (2 = CommonJS)

### The Critical Finding: String Ordering

In the **ORIGINAL** binary, the data region layout for cli.js is:
```
[bytecode at offset 120, 111,477,040 bytes]
[content at offset 111,477,232, 12,791,987 bytes]
```

**Bytecode comes BEFORE content.** The bytecode contains pre-compiled code with absolute offset references back into the source content.

In the **REBUILT** binary (`rebuildBunData()`), strings are always written in this order:
```
stringsData.push(nameBytes, contentsBytes, sourcemapBytes, bytecodeBytes, ...);
```

This puts **content BEFORE bytecode** — the opposite of the original layout:
```
[content at offset ~36, 12,799,553 bytes]
[bytecode at offset 12,799,591, 111,477,040 bytes]
```

### Why 24/24 Worked But 27/27 Doesn't

The bytecode contains hard-coded source position references. When the content moves from offset 111,477,232 to offset ~36 (a massive shift), Bun's bytecode validator detects the mismatch. However:

- At **24/24** (7,133 bytes added): Bun tolerates the layout change. The bytecode may have a validation tolerance, or the specific patched regions don't affect critical bytecode boundaries.
- At **27/27** (7,564 bytes added, 431 more): The additional content crosses a threshold in Bun's bytecode-to-source validation, causing the "Expected CommonJS module to have a function wrapper" error.

This is a **fragile equilibrium** — even at 24/24, the repack was technically producing a non-standard layout that Bun happened to tolerate.

### What Was Tried and Failed

1. **Clearing bytecode** (`bytecodeBytes = Buffer.alloc(0)`): Still crashes. Bun may still expect bytecode when `encoding: 1`.
2. **Clearing bytecode + setting encoding to 0**: Still crashes. The module format or loader flag may also need adjustment.
3. **No-op thinking patches** (patches exist in PATCH_DEFINITIONS but return content unchanged): Still crashes — because the 24/24 patches are also reordered, and the shim auto-reapplied.

### What DID Work

- **Manual `/bin/cp` of backup binary**: Always works. The backup is a clean Anthropic-signed binary.
- **24/24 patches via d980eb0 code**: Works consistently. The repack with 7,133 added bytes is within Bun's tolerance.

## The Fix Plan

### Option A: Preserve Original String Ordering (Recommended)

Modify `rebuildBunData()` to preserve the original string layout order instead of using a fixed order. For each module:
1. Record the original offsets of all string pointers
2. Sort strings by their original offset
3. Write strings in the original order
4. Update StringPointers to new offsets

This ensures bytecode references remain valid because the relative layout matches what Bun compiled.

**Implementation:**
```typescript
// In rebuildBunData, instead of:
stringsData.push(nameBytes, contentsBytes, sourcemapBytes, bytecodeBytes, ...);

// Do:
const moduleStrings = [
  { key: 'name', offset: module.name.offset, data: nameBytes },
  { key: 'contents', offset: module.contents.offset, data: contentsBytes },
  { key: 'sourcemap', offset: module.sourcemap.offset, data: sourcemapBytes },
  { key: 'bytecode', offset: module.bytecode.offset, data: bytecodeBytes },
  // ... moduleInfo, bytecodeOriginPath for new format
];
moduleStrings.sort((a, b) => a.offset - b.offset);
// Push in sorted order, track which StringPointer maps to which index
```

### Option B: Clear Bytecode + Force Source-Only Mode

Set `encoding: 0`, `bytecodeBytes = Buffer.alloc(0)`, and potentially clear `bytecodeOriginPath`. This forces Bun to recompile from source at startup (slower first launch, but works).

**Risk**: The `loader` and `moduleFormat` fields might also need adjustment. Bun's source code would need to be consulted to determine the exact combination that triggers source-only mode.

### Option C: Hybrid — Preserve Layout for Bytecoded Modules Only

Only reorder strings for modules that have bytecode (`bytecodeLen > 0`). For all others, use the current fixed order. This is a targeted fix with minimal risk.
