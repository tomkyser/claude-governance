## Summary

## Key Decisions

### Override Injection Architecture
Instead of patching each message component separately (SystemTextMessage, AssistantToolUseMessage),
discovered that both render through a single dispatcher function `oOY(q)` with a
`switch(_.type)` statement covering all message types. Injected the override check
BEFORE the switch, so a single patch handles all types. Same for content blocks via `sOY(q)`.

### Lazy Override Loading
The message-override patch lazily requires `~/.claude-governance/overrides/defaults.js`
on first render. Uses `globalThis.__govOverridesInit` flag to ensure one-time load.
This keeps the binary patch minimal while enabling extensible runtime overrides.

### Deploy Pipeline Extension
Added override file deployment alongside existing tool/UI/prompt deployment.
`data/overrides/defaults.js` → `~/.claude-governance/overrides/defaults.js`.

## Technical Findings

### Message Rendering Architecture (esbuild 2.1.101)
- `oOY(q)`: Main message renderer — `switch(_.type)` with cases: attachment, assistant,
  user, system, grouped_tool_use, collapsed_read_search
- `sOY(q)`: Content block renderer — `switch(_.type)` with cases: tool_use, text,
  redacted_thinking, thinking
- React ref: `J9`, Box: `u`, Text: `V`
- OPK: Attachment renderer — 28 cases, 10 return null (skill_listing, agent_listing_delta,
  invoked_skills, command_permissions, hook_*, async_hook_response)
- Detection anchors: `_6(94)` for oOY, `_6(48)` for sOY — memo cache sizes are stable

### Build Caching
tsdown/rolldown may cache content chunks. After adding registry entries, a `rm -rf dist`
was required to get entries into the built output. Regular `pnpm build` sometimes used
stale chunk hashes.
